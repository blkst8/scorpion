8. Architecture Notes for Scorpion

  1. Poll interval is latency's hard floor. sse.poll_interval: 200ms → p50 ≈ 183 ms at 1 000 clients. Halving to 100ms halves median latency but doubles Redis ops/sec. Profile Redis before tightening.
  2. SSE scales linearly in goroutines. ~126 000 goroutines at 10 000 clients. Each Go goroutine starts at ≈ 8 KB stack. Always set ulimit -n 65536 and tune GOMAXPROCS for your CPU count.
  3. Redis is the throughput ceiling, not HTTP. At 5 000 clients: 66 685 peak Redis ops/sec, 496 557 total commands for SSE. Pool sizing (pool_size: 2048, min_idle_conns: 256) is critical.
  4. Polling is the natural fallback. When a client cannot hold a persistent SSE stream (proxy timeouts, slow mobile), GET /v1/events/:clientID provides the same Redis-backed queue — same order, just bounded by poll cadence.
  5. MQTT is complementary, not a replacement. MQTT/EMQX excels at IoT and internal microservice messaging. For browser-facing event delivery at scale (> 2 000 users), SSE wins on reliability and zero-miss guarantees.
  6. ACK flow adds at-least-once semantics to SSE. Scorpion's POST /v1/ack path tracks in-flight events in Redis. ACKed events are removed from the inflight registry — without a separate broker.
